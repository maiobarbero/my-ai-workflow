# Planning Rules

Use these rules to turn complexity diagnosis into a practical intervention plan.

## Priority

Prioritize actions by:

1. High severity hotspots.
2. Actions that unblock decisions, approvals, or technical dependencies.
3. Actions that reduce high dynamic uncertainty early.
4. Actions that create shared facts before irreversible commitments.
5. Actions that protect value opportunities.
6. Medium severity hotspots with weak mitigation.

Do not prioritize by TOE score alone.

## Sequencing

Use three horizons:

| Horizon | Meaning |
|---|---|
| Now | Start immediately; clarifies uncertainty, prevents blockage, or creates needed control |
| Next | Start after first facts, decisions, or alignment are available |
| Later | Reassess or execute after phase change, pilot, procurement, or design freeze |

## Action Shape

Each action should include:

- Hotspot
- Intervention type: Control, Connecting, or Mixed
- Concrete action
- Owner role
- Time horizon
- Success signal
- Risk if ignored

Keep actions specific enough to execute, but not so detailed that they become a
project schedule.

## Owner Roles

Use roles, not names, unless names are provided.

Common owner roles:

- Project manager
- Sponsor
- Product owner
- Technical lead
- Integration lead
- Interface manager
- Contract manager
- Legal/compliance lead
- Communications lead
- Stakeholder lead
- Risk manager
- Steering committee

## Controls

Use control actions when detail complexity is high:

- Baseline scope.
- Define interfaces.
- Create decision gates.
- Set acceptance criteria.
- Assign clear owners.
- Maintain risk register.
- Track dependencies.
- Clarify contract responsibilities.

## Connections

Use connecting actions when dynamic complexity is high:

- Align actors on problem framing.
- Run shared tradeoff sessions.
- Build joint scenarios.
- Create transparent escalation path.
- Use short feedback cycles.
- Revisit assumptions at phase gates.
- Surface conflicting incentives.

## Reassessment

Include reassessment triggers when:

- Project phase changes.
- Major stakeholder changes position.
- Scope changes materially.
- A key dependency slips.
- Procurement or contract structure changes.
- External regulation, market, or politics shifts.
- Team composition changes.

## Guardrails

- Do not create generic project management checklist.
- Do not add actions unrelated to diagnosis.
- Do not create stakeholder interview questions.
- Do not promise complexity reduction; some complexity must be managed or used.
- Do not remove interaction from dynamic hotspots.
- Do not remove control from high-detail hotspots.
